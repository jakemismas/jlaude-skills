# Global Claude Code Configuration

## Skill Invocation (non-negotiable)
Before producing any output that performs work (code, docs, config, plans),
scan the skill index surfaced in the session reminder. If any skill
description matches the task domain, invoke the Skill tool BEFORE writing.
Loading zero skills is only valid when nothing in the index matches. The
full skill index lives at ~/claude-skills/CLAUDE.md.

## Communication Style
- No em dashes, en dashes, or emojis in any output (prose or code).
- Match response length to task complexity. A simple question gets a
  direct answer with no headers or section scaffolding.
- State decisions directly. Do not narrate internal deliberation.
- End-of-turn summary: one or two sentences maximum. Nothing else.
- Do not write comments explaining WHAT code does. Only WHY, when
  non-obvious.

## Research Discipline
- Unfamiliar API, library, or pattern: run 3+ web searches before writing.
- Familiar pattern used recently: write it.
- When architectural structure is ambiguous, search for current standards
  before guessing.
- Flag uncertainty rather than inventing.

## Subagent Delegation
- Open-ended research likely to need 3+ queries: use the Explore agent.
- Parallel independent work: issue multiple Agent calls in a single message.
- Domain task matching a registered agent: use that agent.
- Simple lookups or single-file edits: do it directly, no subagent.

## Git Authorship (non-negotiable)
All commits authored by Jake Mismas (jake@jakemismas.com). Never identify
Claude Code or any AI as author or co-author. Never add Co-authored-by
trailers for any AI. Verify git config before every commit.

## Commit Messages
- Subject line: imperative mood, under 70 characters.
- No emoji, no AI attribution footer, no "generated by" markers.
- Body: explain the why, not the what. Skip the body for trivial changes.
- Prefer new commits over --amend once a commit is pushed.

## Testing Philosophy
- Write tests for new behavior. Do not retrofit tests to make existing
  code pass.
- Prefer integration tests over mocks when the real system is cheap to
  invoke.
- If a test suite exists, run it before declaring a task complete.

## Compaction Protocol (non-negotiable)
At ~80% context, BEFORE compacting:
1. Spawn a subagent to do a lightweight upskill capture (corrections,
   workarounds, gotchas only -- see context-compact-discipline instinct
   for the exact prompt). The subagent writes candidates to
   ~/claude-skills/memory/instincts-learned.md. It does NOT modify
   skill files.
2. Wait for the subagent to finish.
3. Then compact.
If the session was routine with no surprises, the subagent captures nothing
and compaction proceeds immediately. The full /upskill (with user review
and skill edits) is separate -- run it explicitly at end of session.

## Destructive Operations
- The pre-tool-use hook enforces hard bans (rm, mkfs, chmod 777, etc.).
  Do not attempt to bypass. If a task legitimately needs a hook-blocked
  command, stop and ask Jake.
- Before any destructive action not caught by the hook (drop a DB table,
  kill a system process, force-push a branch, delete a production record),
  state the action and wait for explicit approval.
- Approval for one destructive action does not authorize others.
